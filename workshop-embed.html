AWS Shared Responsibility Model

Before diving into Prowler, let's review the AWS shared responsibility model and fundamental AWS security principles. It's crucial to understand:

  • AWS Responsibility: Security "of" the cloud (infrastructure, hardware, software, networking)
  • Customer Responsibility: Security "in" the cloud (data, identity management, applications, operating systems, network configuration)

Prowler focuses on helping you secure your responsibilities within the shared model.

What is Prowler?

Prowler is the world's most widely adopted, open, AI-driven cloud security platform that detects misconfigurations, prioritizes risk, accelerates remediation, and ensures continuous compliance.

Setting Up Prowler

1

Sign Up for Prowler Cloud Trial

Sign up for a Prowler Cloud account by clicking here. Be sure to provide a valid email address as a verification code will be sent to the email address and choose a password that conforms to the requirements.

2

Verify Your Account

Verify your account by typing in the code from the email that you received from Prowler. If you don't find the email in your Inbox, be sure to check the Spam folder.

Note: If you choose Google or GitHub for registration, then it will not ask you to verify your email.
3

Login to Prowler Cloud

Login to Prowler Cloud using the credentials you used above. You will see the Overview page with no details as we have not connected your cloud account just yet. We will do that next.

4

Connect Your AWS Account

Now that you have a Prowler account, you'll notice the Overview page doesn't show meaningful insights yet. This is because we haven't connected your AWS account to Prowler Cloud. Let's connect your AWS account and run your first security scan.

To navigate to Cloud Providers:

  1. In your Prowler Cloud dashboard, look at the left-hand navigation menu
  2. Click on Overview under Analytics (you may need to expand this section)
  3. Under Providers Overview, click on Add Provider

This will take you to the Provider list page, and if you don't have connected providers, it will be empty. Click Add Cloud Provider on this screen.

From the list of providers choose Amazon Web Services.

In the next screen, you'll need to enter your AWS account details:

To find your AWS Account ID:

  1. In your AWS Console (separate tab), look at the top-right corner next to your username
  2. Your 12-digit Account ID is displayed there (e.g., 123456789012)
  3. Copy this number

To complete the connection:

  • Account ID: Paste your 12-digit AWS Account ID
  • Provider Alias (optional): Enter a friendly name like "Workshop Account" to help identify this account later
  • Click the Next button to proceed
Tip: Adding an account name is helpful when you have multiple AWS accounts connected to Prowler.

There are two ways to connect Prowler to your AWS account:

  1. Connect Assuming IAM Role (Recommended)
  2. Connect using (temporary) credentials

For this workshop, we will connect assuming IAM Role. This provides an enduring authentication for Prowler to run checks at regular intervals. The default scan frequency is daily, but you can configure checks to run at any predetermined frequency.

5

Create IAM Role with CloudFormation

In this step, you will create a read-only IAM Role using the provided CloudFormation template in Prowler UI.

To create the IAM Role:

  1. Click on "Use the following AWS CloudFormation Quick link to create the IAM Role" in the Prowler interface
  2. This will open the CloudFormation template in a new AWS Console tab
Important: Keep the Prowler page open — you'll need to return to it to enter the ARN of the newly created IAM Role.

CloudFormation Deployment Steps:

  1. The template will pre-populate with the necessary IAM permissions for Prowler
  2. Keep all default settings — do not modify any parameters or configurations
  3. Check the box under "Capabilities" to confirm that CloudFormation can create IAM resources
  4. Click "Create Stack" to deploy the role
  5. Wait for the stack creation to complete (usually takes 1-2 minutes)
  6. Once the stack status shows "CREATE_COMPLETE", navigate to the Resources tab in CloudFormation
  7. Click on ProwlerScan under the Physical ID column — this will open the IAM role details
  8. Copy the Role ARN from the IAM role details page
  9. Return to the Prowler tab and paste the ARN to complete the connection
6

Complete Connection and Launch First Scan

Validate the connection: When you click Next, Prowler will validate the connection and add this AWS Account to your Providers list.

Congratulations! You've successfully connected your AWS account to Prowler Cloud and are ready to start your security assessment.

Hands-On Remediation Exercise

1

Analyze Critical S3 Finding

Click on the critical finding that Prowler is recommending for Amazon S3.

When you click on the info icon next to the finding on the left, it opens up the details drawer. Go through the description and risk. The finding details page also links to specific service documentation to learn more about that specific resource and misconfiguration.

2

Apply Manual Remediation

The finding details will also provide a CLI command to remediate the issue. We'll use AWS CloudShell to execute this command.

To remediate using CloudShell:

  1. Open CloudShell: In the AWS Console, you can open CloudShell in two ways:
    • Option 1: Type CloudShell in the search bar at the top of the console and select it from the results
    • Option 2: Click on the CloudShell icon at the bottom left of the console (it looks like a terminal/command prompt icon)
  2. Copy the remediation command from the finding details page, or use the command below and paste in CloudShell
  3. Update the bucket name with your actual bucket name from the finding details
Bash 
1aws s3api put-public-access-block \
2  --public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true \
3  --bucket public-bucket-central-whippet

Execute the command in CloudShell by pressing Enter.

Note: Replace public-bucket-central-whippet with your actual bucket name from the finding details.

Understanding the Remediation Command

This command will apply a Public Access Block configuration on the specified S3 bucket. Let's break down exactly what it does:

Purpose: This command prevents any kind of public access to the specified S3 bucket by setting all four "Public Access Block" settings to true.

These settings are bucket-level security controls that override other ACLs and bucket policies that might otherwise make the bucket or its objects public.

You have now remediated the finding. The next time Prowler runs a scan on your cloud resources it will not find a publicly accessible S3 bucket.

Working with Mutelists

Mutelist is a feature in Prowler that lets you selectively mute (suppress) findings that you intentionally accept or that are otherwise expected deviations from best practices. It gives you fine-grained control over which checks, resources, accounts, regions, or tagged items should be excluded from alerts or reports.

Creating a Mutelist

The configuration is defined in a YAML file, where you can specify combinations of account, check, region, resource, and tags, using exact strings or regular expressions.

Hands-On Exercise: Create a Mutelist

Create a mutelist that will suppress findings on check id ec2_instance_port_ssh_exposed_to_internet for the EC2 instance you have created for git_org=awsworkshop.

Review the following YAML structure and add it to Mutelist — left navigation > Configuration > Mutelist

YAML 
 1Mutelist:
 2  Accounts:
 3    '*':
 4      Checks:
 5        ec2_instance_port_ssh_exposed_to_internet:
 6          Tags:
 7            - git_org=awsworkshop
 8          Regions:
 9            - '*'
10          Resources:
11            - '*'

What this mutelist does:

  • Applies to all accounts (*)
  • Mutes the EC2 SSH exposure check specifically
  • Only for resources tagged with git_org=awsworkshop
  • Covers all regions and resources with wildcards

To validate that the finding is suppressed, initiate a new scan by clicking on Launch Scan button on the left navigation, and selecting the Provider from the drop-down menu. Then click on Start Now to begin scanning. After the scan is complete, verify that critical finding is suppressed or no longer reported by clicking on See Findings under the Findings column.

To confirm that Prowler found the misconfiguration but is muting it, click on Browse All Findings in the left navigation, and look for the finding with the muted icon.

Supported Compliance Frameworks

Prowler offers 36 out-of-the-box compliance frameworks for workloads and services deployed on AWS. You can refer to the compliance frameworks, their requirements and checks here.

To explore the available compliance frameworks and their specific requirements:

  1. Visit Prowler Hub
  2. Click on Browse Security Compliance Frameworks and select AWS

Running Compliance Assessments

As Prowler scans cloud resources and generates findings, it also maps those findings to requirements of each compliance framework and prepares details on key gaps.

Let's walk through how to access and review these compliance assessments:

  1. Click on Compliance in the left navigation
  2. Choose any compliance framework of your interest to understand key gaps and failing requirements. For example, the report for AWS Foundational Security Best Practices
Note: You may have some requirements that are not applicable to your cloud environment in a certain compliance framework simply because you may not have all pertinent resources provisioned.

As you work with these compliance reports, you can improve compliance score as you remediate findings and create an evidence package for auditors to prove that you meet the requirements for desired frameworks.

Introduction to Lighthouse AI

Lighthouse AI is a collection of Agentic AI tools designed to simplify cloud security and compliance and dramatically improve Security Operations efficiency by directly integrating into their workflows.

Lighthouse AI Configuration

You should have already received the OpenAI key in an email to complete the next section.

To configure Lighthouse AI:

  1. Click on Configuration in the left navigation bar
  2. Click Lighthouse AI
  3. Choose the GPT model (GPT-4o is recommended)
  4. Enter your OpenAI API Key (You should have already received the OpenAI key in your email.)
  5. Click Save

Once configured, you'll be able to use the Lighthouse AI-powered analysis for your security findings.

Using Lighthouse AI for Security Analysis

You can ask Lighthouse AI questions in natural language as you analyze security findings and potential vulnerabilities. Recall, we had a finding on AWS Lambda IAM policy that could lead to potential privilege escalation. Now let's ask Lighthouse AI about this specific security concern:

Lighthouse AI Prompt 
Do I have an IAM policy in my AWS account that could lead to privilege escalation?

Copy and paste this question into the Lighthouse AI text box. Lighthouse AI will provide a detailed explanation of the finding, including the associated risk and how someone could potentially exploit these permissions for privilege escalation.

Once you understand the security issue, you can take it a step further and ask Lighthouse AI to help create a solution:

Lighthouse AI Prompt 
Can you create an inline policy that I can replace excess_policy in AWS Management Console to eliminate risk of privilege escalation with AWS Lambda, render the policy as a codeblock?

In response, Lighthouse AI will generate an appropriate inline policy along with step-by-step instructions on how to update the existing policy using the AWS Management Console.

Hands-On Exercise

Practice using Lighthouse AI by asking the questions below. Access Lighthouse AI in your Prowler Cloud dashboard, then copy and paste each question into the AI text box and explore the responses:

1. Show me the most common misconfigurations in my AWS account and how to fix them.

2. Which storage buckets in my AWS account are missing encryption at rest?

2

Delete CloudFormation Stack

  3. Click Delete stack to confirm
